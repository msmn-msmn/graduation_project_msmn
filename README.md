# ■サービス概要
スケジュール管理アプリ<br>
スケジュールを管理するのではなく管理できるスケジュールを作ることを目的とする。<br>
計画倒れにならない実現可能な計画立案のサポートをする。<br>
<br>
<br>

# ■ このサービスへの思い・作りたい理由
自分の困っていることを自己分析により抽出した結果<br>
```
・工数の見積もりが苦手
・ギリギリまで先延ばしにする
・締め切り間近に慌てる
・悩んで行動に移せない
```

という項目を抜き出しました。<br>

これの原因をフローで考えると、<br>
＜工数の見積もりの見積もりが苦手＞
```
全体像の把握が不十分 → タスク分解の粒度が粗い → 何をすればいいかわかってない → 見積もりとバッファ不足
```

＜ギリギリまで先延ばしにする・締め切り間近に慌てる・悩んで行動に移せない＞
```
見積もりとバッファ不足 → そのまま計画→ いつまでにどこまでの認識の欠如
→ まだ間に合うと先延ばし or 行動に移していいか悩む → 計画の取り掛かりが遅延する
→ 計画のフィードバックをしない → 計画の遅延が数値的に認識できない → 締切間近で遅延を認識する
→ 計画倒れ or 何とか間に合わせる
```

というように分解しました。<br>
<br>
開発においても普通に作業するにしても時間管理は重要です。<br>
でも段取りが分からない・大変そう or まだ間に合うとつい先送りしてしまう等でスケジュールが当初の予定より<br>
遅延することをよくしてしまいます。<br>
<br>
現状も卒業制作に取り掛かる時期が当初の予定よりも大幅にずれ込んで非常に焦っています。<br>
そこで、タスク分解をサポートして管理できるスケジュールが作れるアプリがあったら<br>
最高だなと思い卒業制作のテーマにしました。<br>
<br>
<br>

# ■ ユーザー層について
スケジュールを立てるのが苦手な社会人をメインターゲットにしています。<br>
理由としては、<br>
```
・現状の自分が該当すること
・書店のビジネス所の棚にスケジュール管理のいろんな種類の啓発本が多数並んでいることから
　スケジュール管理で悩む社会人は多く、需要がある
```
と考えたからです。<br>
<br>
<br>



# ■サービスの利用イメージ
## ・サービスの流れ
```
ユーザーがタスク登録　→　小タスクに分解をサポート　→　小タスクをタスクのマイルストーンに設定
一定期間ごとに計画の見直しが発生　→　計画の修正　→　計画達成！！
```
<br>

## ・ユーザー価値
タスクを分解することで全体像を把握して、マイルストーンの設置と計画見直しタスクで締め切りを見える化することで
計画倒れ・炎上を防止する。
<br>
<br>

## ・使用例（タスク：READMEのLGTMをもらう）
### 1．ユーザーは最初にタスク名を入力する（タスク：READMEのLGTMをもらう）

### 2．アプリの入力内容からAIがWBS法でタスクを段階ごとに分解する（WBS法は変更するかもしれません）
タスクが細かく分解されることでユーザーの全体像に対する解像度を上げて<br>
何をしたらいいか漠然とした状態になることを減らします。<br>
タスクが分解された各段階ステップがマイルストーンとして機能します。<br>
本リリースではステップごとに看板形式にされた状態で表示されます。<br>

分解イメージ.<br>
```
READMEのLGTMをもらう
├─ 1. READMEドラフト作成
│   ├─ 1.1 プロジェクト概要を書く
│   ├─ 1.2 セットアップ手順を書く（Docker、Railsなど）
│   ├─ 1.3 使用技術とバージョンを明記する
│   ├─ 1.4 画面キャプチャやGIFを挿入する
│   └─ 1.5 その他（ライセンス・補足情報など）を書く
├─ 2. チェック＆推敲
│   ├─ 2.1 文法・誤字の確認
│   ├─ 2.2 情報の抜け・分かりづらさを確認
│   └─ 2.3 表現のトーンを統一する（丁寧/カジュアル）
├─ 3. フォーマット調整
│   ├─ 3.1 Markdown記法の整形（見出し、リストなど）
│   ├─ 3.2 コードブロック・リンクの確認
│   └─ 3.3 画像リンクやGIFの表示確認
├─ 4. GitHubにPush
│   ├─ 4.1 READMEをステージング・コミット
│   └─ 4.2 GitHubにPushし、反映を確認
├─ 5. フィードバック依頼
│   ├─ 5.1 レビュワー（メンター・チームメイト）を選ぶ
│   ├─ 5.2 PR or Issueを立ててレビュー依頼
│   └─ 5.3 「LGTMください！」と丁寧にメッセージ送信
└─ 6. フィードバック対応・最終反映
    ├─ 6.1 指摘事項の修正
    ├─ 6.2 修正後の最終Push
    └─ 6.3 LGTM確認、SlackやGitHubで完了報告
```
<br>

### 3.ユーザーが分解結果に対して編集する
ステップや内容の変更<br>
いらないステップの削除または必要ステップの追加<br>
<br>

### 4.時間設定をする
タスクの締切を設定<br>
タスクの締切までのユーザーのタスクに使える可処分時間を入力する（日毎、週毎）<br>
ステップごとの必要時間と締切を設定<br>
必要時間に対して可処分時間が少ない場合警告が出ることでタスクに対してバッファがないことを見える化する。<br>
（制約として１週間以内なら可処分時間の日割り、１週間より長期なら週毎の可処分時間の80％までしか<br>
期間ごとの必要時間に設定できない感じ）<br>
<br>

### 5.タスクを登録する
設定が終わったタスクを登録して保存します。<br>
登録後は小タスクを消化したらチェックを入れていく形で進行していきます。<br>
<br>

### 6.振り返りタスクの発生
締め切りまでの期間が１週間よりも長期のタスクは週ごとに進捗確認タスクが発生することで予定に対して現状が<br>
どこまで進んでいるかをユーザーにフィードバックして遅延しているか順調なのかを<br>
認識してもらい計画の修正が必要か判断してもらいます。<br>
定期的にPDCAサイクルを回すタスクを発生することでユーザーに計画を立てた後に<br>
見直しをしないということを減らします。<br>
<br>
<br>

# ■ ユーザーの獲得について
## ＜メインプラットフォーム＞
メインプラットフォームをPCとしてサービスを提供したいと考えています。<br>
理由としては、タスク編集のしやすさによるユーザビリティの向上と早期開発のためです。<br>

## ＜周知の方法＞
最初にRUNTEQ内のMattermostやポモドーロ休憩チャットで告知してフィードバックを集めます。<br>
次にフィードバックを内容を取り入れたものをnote等の技術記事で発信して更にフィードバックを集めます。<br>
最後にSNSで発信して周知をしながら改修を進めようと思っています。<br>

## ＜そうした理由＞
メインターゲットとする***スケジュールを立てるのが苦手な社会人***について考えると<br>
世代的には10代後半～50代位の範囲で特に10代後半～40代位がボリューム層に当たると考えました。<br>
そうした世代の集まりやすい場所はSNSやnote等のネットで無料でスケージュール管理方法を見れる場所だと<br>
考えてアプリの機能がある程度固まるごとに徐々に範囲を広げる形で告知をしたいと考えました。<br>
<br>
実際にnoteでスケジュール管理で検索すると<br>
```
#スケジュール (14,291)
#タスク管理 (10,929)
#手帳 (41,751)
#日記 (4,766,770)
#時間管理 (13,943)
#毎日note (1,555,535)
```
上記のように多数の検索結果が出ました。<br>
見るだけの人はそれ以上になると考えられるのでここから誘導することが出来たらその口コミを記事等で<br>
シェアしてくれるかもしれないとも考えました。<br>

<br>
<br>

# ■ サービスの差別化ポイント・推しポイント
似たようなサービスが存在する場合、そのサービスとの明確な差別化ポイントとその差別化ポイントのどこが優れているのか教えてください。
独自性の強いサービスの場合、このサービスの推しとなるポイントを教えてください。
## ・Githubのissueのボード機能		
**差別点：簡単な操作**<br>
端的にいうと、導入が簡単であることです。<br>
Githubのissueのボード機能は細かい設定や自動化を行うことでプロジェクトの<br>
管理をカスタマイズし、チーム連携をすることが出来て操作方法になれると出来ることが広がるツールです。<br>
しかし、操作方法は慣れていない人には設定が難しく、用途もソフトウェア開発向けで一般の用途には<br>
正直むいてないです。<br>
<br>
個人的な感想ですが、使いこなせれば便利という有名アプリは色々とありますが、"使いこなせれば"という所で<br>
導入を諦めている人も多いのではないでしょうか？<br>
本アプリは、機能をシンプルにすることでカスタム性が少ない代わりに操作方法を勉強する手間を<br>
大幅にカットしています。<br>
<br>

## ・Trello				
**差別点：立てた計画の管理しやすさではなく、管理しやすい計画を立てることを目的とする**<br>
端的にいうと、主要サービスの開始タイミングが計画をする前に始まるか計画をした後に始まるかの違いになります。<br>
<br>
Trelloは画面遷移が少なく直感的に扱えてカスタマイズも豊富なとても良いアプリだと思います。<br>
しかし、あくまでも計画をした後の管理がしやすいのであって大本の計画自体に無理があると<br>
アプリの良さを生かしきれません。<br>
<br>
本アプリでは計画を立てる段階からサービスの提供を開始して目的達成のためにユーザーが<br>
何をしたら良いのかを明確化し、ユーザーがタスクに割ける時間よりも多くの必要時間を設定することを<br>
禁止することでバッファを持った無理のない計画立案を補助します。<br>
<br>
<br>
<br>
<br>

# ■ 機能の実装方針予定
## ＜開発環境＞
```
使用言語：　　　　Ruby 3.3.6
フレームワーク：　Ruby on Rails　7.2.2
DB：　　　　　　　PostgreSQL　16.8
デプロイサーバー：Render
```
<br>
<br>

## ＜MVP機能＞
### ・ユーザー登録機能とログイン/ログアウト機能
```
gem 'devise' 4.0
gem 'omniauth' 2.1.2
gem 'omniauth-google-oauth2'
gem 'omniauth-rails_csrf_protection'
```
とActionMailer でサインインとパスワードとパスワードリセット機能を実装<br>
改修でログイン手段を徐々に増やす予定（Twitter、Facebook等）<br>
<br>

### ・タスク　→　小タスク分解機能
```
gem 'ruby-openai' 5.1.0
```
とOpenAI API で入力内容から小タスクに分解した結果をユーザーが編集できる機能を実装<br>
<br>

### ・計画の振り返り機能
```
gem 'ice_cube' 0.17.0
gem 'ultimate_turbo_modal' 2.0.0
```
で指定期間ごとに振り返りタスクのモーダルが発生する機能を実装<br>
<br>
<br>

## ＜本リリース機能＞
### ・ユーザーの可処分時間に対してのタスクの占有率表示機能
```
gem 'Bootstrap' 5.3.0
```
を使って進捗バー形式で占有率を表現する<br>
<br>

### ・タスクを看板形式で表示機能
 JavaScript ライブラリ Sortable.jsを導入することでタスク画面の表示を看板形式に実装予定<br>
<br>

### ・リマインド機能
ActionMailer + Active Jobで設定日時にメールでお知らせ<br>
<br>

### ・タスク検索機能
```
gem 'ransack' 4.3.0
gem 'kaminari'
```
で検索機能、ページネイトを実装<br>
<br>
<br>
